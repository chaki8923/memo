お疲れ様です。本日は、私たちIBJサイトチームの今後のあり方について、重要な指針を共有させてください。 テーマは**「実装者から、価値の共創者へ」**です 。

これまで私たちは、仕様書通りに正しく動くものを作る「実装者（Implementer）」として素晴らしい仕事をしてきました。しかし、これからの時代、そしてチームがさらに成長していくためには、もう一段階上のステージへ進化する必要があります 。今日はその「進化の方向性」についてお話しします。

【なぜ変わるのか？】
[Slide 2: VISION & MISSION]

まず、私たちのビジョンとミッションについてです。 一言で言うと、**「コードを書くことを超える」**ということです 。

皆さんも感じている通り、今は「AI時代」です。単にプログラムのコードを書くだけなら、AIがサポートしてくれる時代になりました 。では、私たちエンジニアの本質的な価値はどこに残るのでしょうか？

それは、「プログラムの実装」そのものではなく、**「ビジネスの課題を技術で解決し、持続可能なシステムを設計すること」**です 。 これからは、ただ言われたものを作る「作り手」ではなく、ディレクターやビジネスサイドと共に事業価値を創り上げる「共創者」へと進化していきましょう 。


【3つの重要な指針】
では、具体的にどう変わるべきか。大きく3つのポイントがあります。

[Slide 3: 01. 数字と顧客に向き合う]

1つ目は、**「数字と顧客に向き合う」**ことです 。 厳しい言い方になるかもしれませんが、機能開発そのものはゴールではありません。それはあくまで目標達成のための「手段」です 。


コードを書き始める前に、一度手を止めて考えてみてください。 「これでUXはどう良くなるのか？」「どの数字（KPI）に効くのか？」 。 これからは、期待される成果を一言で言語化してから着手する。この習慣を徹底しましょう 。



[Slide 4: 02. プロとしての設計力]

2つ目は、「プロとしての設計力」です 。 先ほど、コーディングはAIが助けてくれると言いましたが、「全体観」を持った設計はAIにはまだ真似できません 。


私たちが責任を持つべきは「品質の土台」です 。 システム全体のアーキテクチャ、堅牢なセキュリティ、そして将来を見据えた拡張性 。こここそが、プロフェッショナルとしての腕の見せ所になります。


[Slide 5: 03. 未来への投資を怠らない]

3つ目は、**「未来への投資を怠らない」**ことです 。 よくある「余裕ができたらリファクタリングする」という言葉、これからは禁止にしましょう 。なぜなら、余裕ができる日なんて永遠に来ないからです。


面倒なコード整理や環境改善は、特別なイベントではなく、日々の仕事の一部として当たり前に組み込んでください 。 常に足場を固めることが、結果として将来の開発スピードを加速させます 。


【具体的なアクションプラン】
では、明日から具体的に何をするのか。3つのアクション（A, B, C）を定義しました。

[Slide 6: ACTION A ビジネスへの貢献]

まずAction Aは、ビジネス貢献のアプローチです。 チケットに着手する前、「なぜやるのか」を1行でいいので記述してください 。誰のどんな悩みを解決するのか、クリアにしてから走り出しましょう。

また、「これ作って」と依頼を待つのではなく、データに基づいて「ここがボトルネックだから直したい」と、エンジニアから提案するスタイルへ変えていきます 。月に1回、自分たちでサービスを使い倒して改善点を出し合う会も実施します 。


[Slide 7: ACTION B 品質の向上]

次にAction B、品質向上とAI活用です。 コードレビューのやり方を変えます。細かい記法やスタイルのチェックはAIやLinterに任せましょう 。 人間が議論すべきは、**「設計」**です。「設計思想は正しいか？」「将来変更しやすいか？」といった本質的な議論に時間を使いましょう 。



また、単に機能通り動くだけでなく、アクセス集中への耐性やセキュリティなど、非機能要件を含めた「品質チェックリスト」を基準にします 。


[Slide 8: ACTION C 足場固めと環境改善]

最後にAction C、環境改善です。


「来た時よりも美しく」ルールを導入します 。 機能追加のついでに周辺コードをリファクタリングする。これを見積もりに含めることをチームとして推奨します 。


また、金曜の午後などを「改善タイム」として公式に確保し、機能開発以外の負債解消に充てます 。開発のしやすさ、つまり「待ち時間」や「手戻り」を減らすことも、皆さんの評価の一部として大切にしていきます 。


【目指す姿とまとめ】
[Slide 9: 成功の定義 (OUTCOME)]

これらを実行することで、私たちはどんなチームになるのか。 定量的な指標としては、DORA指標（リリース頻度など）の向上や、技術的負債の抑制を目指します 。


しかし、もっと重要なのは定性的な変化、つまり「会話の質」が変わることです 。 「何を作るか（What）」だけの会話から、 「なぜやるのか？（Why）」 「どう測るのか？（How）」 「どう安全に運用するか？」 といった、本質的な問いが飛び交うチームになること 。これが私たちの目指すゴールです。
